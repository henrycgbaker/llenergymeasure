## Summary

Phase 19.2 extends the `VLLMConfig` schema with new sub-models and scalar fields, and introduces a universal passthrough architecture (`extra="allow"`) across all backend configs. This is primarily a schema + wiring phase: no new external dependencies, no changes to experiment dispatch, study runner, or Docker infrastructure.

The work decomposes into four coherent areas: (1) passthrough architecture — change `extra="forbid"` to `extra="allow"` on all backend models and forward `model_extra` at runtime, (2) new VLLMEngineConfig fields — 7 scalar fields plus the `VLLMAttentionConfig` nested sub-model, (3) new VLLMBeamSearchConfig — a third top-level nested section under VLLMConfig with beam search wiring, and (4) SSOT updates — introspection coverage and mutual exclusion metadata for the new fields.

The `_build_llm_kwargs()` and `_build_sampling_params()` methods in `core/backends/vllm.py` are the primary runtime touch-points. Both already use the None-skip pattern (`if v is not None: kwargs[k] = v`) which makes adding new fields mechanical. The new beam search path introduces a conditional: when `vllm_cfg.beam_search` is set, construct `BeamSearchParams` instead of `SamplingParams`. All tests are unit-only (no GPU required) — the existing `_FakeSamplingParams` pattern extends naturally.

**Primary recommendation:** Schema → wiring → tests in two plans. Plan 01 covers schema changes (all models, all new fields, extra="allow", cross-validators). Plan 02 covers runtime wiring (beam search path, `model_extra` forwarding for all backends, SSOT update).

### Pattern 1: Pydantic extra="allow" and model_extra forwarding

**What:** Change `model_config = {"extra": "forbid"}` to `model_config = {"extra": "allow"}` on each backend model. Pydantic v2 stores unknown fields in `model_extra` (a `dict[str, Any]`). At runtime, merge these into the kwargs dict.

**When to use:** After all explicit field wiring, before returning kwargs.

**Example:**
```python
# In _build_llm_kwargs():
# After all explicit engine fields have been applied...
if engine.model_extra:
    kwargs.update(engine.model_extra)
```

This pattern applies in:
- `_build_llm_kwargs()` — after explicit engine field iteration, merge `engine.model_extra`
- `_build_sampling_params()` — after explicit sampling field iteration, merge `sampling.model_extra`
- New beam search builder — after explicit beam search field iteration, merge `beam_search.model_extra`

**Pydantic v2 model_extra behaviour (HIGH confidence — verified in codebase):**
- `extra="allow"`: unknown fields accepted, stored in `model.model_extra` as `dict[str, Any]`
- `extra="forbid"` (current): unknown fields raise `ValidationError`
- `extra="ignore"`: unknown fields silently dropped (not what we want)
- When `model_extra` is empty (no extra fields set), it returns `{}` — `if engine.model_extra:` guards correctly

### Pattern 2: None-skip field iteration for new scalar fields

**What:** The existing `_set(k, v)` helper in `_build_llm_kwargs()` already implements the None-skip pattern. New scalar fields follow this exactly.

**When to use:** For every new explicitly-typed field.

**Example (adding to existing `_build_llm_kwargs()`):**
```python
def _set(k: str, v: Any) -> None:
    if v is not None:
        kwargs[k] = v

# New fields (same pattern as existing):
_set("disable_custom_all_reduce", engine.disable_custom_all_reduce)
_set("kv_cache_memory_bytes", engine.kv_cache_memory_bytes)
_set("offload_group_size", engine.offload_group_size)
_set("offload_num_in_group", engine.offload_num_in_group)
_set("offload_prefetch_step", engine.offload_prefetch_step)
_set("compilation_config", engine.compilation_config)

# offload_params: list[str] → set[str] at wiring time (vLLM expects set[str])
if engine.offload_params is not None:
    kwargs["offload_params"] = set(engine.offload_params)
```

Note: `offload_params` is the one field requiring a type conversion. `list[str]` is used in config/YAML because YAML sequences parse to lists. At wiring time, convert to `set[str]` to match vLLM's API. This is the correct choice (Claude's discretion item).

### Pattern 3: VLLMAttentionConfig nested under engine

**What:** New Pydantic model nested as `engine.attention`. Fields are wired into `LLM()` kwargs at the same level as other engine fields (not as a nested dict) — most vLLM attention params are top-level `LLM()` kwargs, not a nested `AttentionConfig` object at the Python API level.

**IMPORTANT:** The CONTEXT.md shows `AttentionConfig` as vLLM's internal type, but at the `LLM()` constructor level, these are passed as flat kwargs. Verify at wiring time which params go directly to `LLM()` vs. need to be wrapped. Given `extra="allow"` passthrough on `VLLMAttentionConfig` itself, any non-wired fields flow through `model_extra`.

**Example schema:**
```python
class VLLMAttentionConfig(BaseModel):
    model_config = {"extra": "allow"}

    backend: str | None = Field(default=None, description="Attention backend: flash_attn, flashinfer, etc.")
    flash_attn_version: int | None = Field(default=None, description="...")
    flash_attn_max_num_splits_for_cuda_graph: int | None = Field(default=None, description="...")
    use_prefill_decode_attention: bool | None = Field(default=None, description="...")
    use_prefill_query_quantization: bool | None = Field(default=None, description="...")
    use_cudnn_prefill: bool | None = Field(default=None, description="...")
    disable_flashinfer_prefill: bool | None = Field(default=None, description="...")
    disable_flashinfer_q_quantization: bool | None = Field(default=None, description="...")
    use_trtllm_attention: bool | None = Field(default=None, description="...")
    use_trtllm_ragged_deepseek_prefill: bool | None = Field(default=None, description="...")
```

**Wiring approach:** Pass attention fields as flat kwargs to `LLM()`. The `backend` field maps to `attention_backend` kwarg (if non-None). The boolean flags map to same-named kwargs. Forward `attention.model_extra` too.

### Pattern 4: VLLMBeamSearchConfig — conditional inference path

**What:** When `vllm_cfg.beam_search is not None`, construct `BeamSearchParams` instead of `SamplingParams`. `BeamSearchParams` was introduced in vLLM v1 when beam search was separated from `SamplingParams`.

**Example:**
```python
@staticmethod
def _build_sampling_params(config: ExperimentConfig, sampling_params_cls: Any) -> Any:
    vllm_cfg = config.vllm

    # Beam search path: use BeamSearchParams instead of SamplingParams
    if vllm_cfg is not None and vllm_cfg.beam_search is not None:
        return VLLMBackend._build_beam_search_params(config, vllm_cfg.beam_search)

    # ... existing SamplingParams path ...

@staticmethod
def _build_beam_search_params(config: ExperimentConfig, beam_cfg: Any) -> Any:
    """Build vllm.BeamSearchParams from VLLMBeamSearchConfig."""
    from vllm import BeamSearchParams

    kwargs: dict[str, Any] = {}
    if beam_cfg.beam_width is not None:
        kwargs["beam_width"] = beam_cfg.beam_width
    if beam_cfg.length_penalty is not None:
        kwargs["length_penalty"] = beam_cfg.length_penalty
    if beam_cfg.early_stopping is not None:
        kwargs["early_stopping"] = beam_cfg.early_stopping
    # max_tokens: override config.max_output_tokens if set
    kwargs["max_tokens"] = beam_cfg.max_tokens or config.max_output_tokens
    # Passthrough
    if beam_cfg.model_extra:
        kwargs.update(beam_cfg.model_extra)
    return BeamSearchParams(**kwargs)
```

**IMPORTANT:** When beam search is used, `llm.generate()` signature changes — `BeamSearchParams` is passed differently from `SamplingParams`. In vLLM v1, `llm.beam_search()` is a separate method. This requires a conditional in `_run_measurement()` as well. See "Open Questions" section.

### Pattern 5: Cross-validators for new mutual exclusions

**What:** Two new cross-validators follow the existing `validate_quantization()` pattern.

On `VLLMConfig` (mutual exclusion: beam_search + sampling):
```python
@model_validator(mode="after")
def validate_beam_search_exclusive(self) -> VLLMConfig:
    if self.beam_search is not None and self.sampling is not None:
        raise ValueError(
            "Cannot use both beam_search and sampling sections simultaneously. "
            "beam_search uses BeamSearchParams; sampling uses SamplingParams."
        )
    return self
```

On `VLLMEngineConfig` (mutual exclusion: kv_cache_memory_bytes + gpu_memory_utilization):
```python
@model_validator(mode="after")
def validate_kv_cache_memory(self) -> VLLMEngineConfig:
    if self.kv_cache_memory_bytes is not None and self.gpu_memory_utilization is not None:
        raise ValueError(
            "kv_cache_memory_bytes and gpu_memory_utilization are mutually exclusive. "
            "Use one or the other to control KV cache memory."
        )
    return self
```

### Pattern 6: SSOT updates for new fields

**What:** `introspection.py` has three functions that need updating:
1. `_get_custom_test_values()` — add invalid values for new constrained fields (`beam_width`, `kv_cache_memory_bytes`, `n`)
2. `get_mutual_exclusions()` — add `kv_cache_memory_bytes` ↔ `gpu_memory_utilization` and `beam_search` ↔ `sampling`
3. `get_backend_specific_params()` — add new explicitly-typed paths to the `vllm` list

The `get_params_from_model()` introspection function already recurses into nested Pydantic models via `hasattr(annotation, "model_fields")`. The new `VLLMAttentionConfig` and `VLLMBeamSearchConfig` will be automatically discovered without any changes to the traversal logic.

### Anti-Patterns to Avoid

- **Don't add `extra="allow"` to `ExperimentConfig` or `VLLMConfig` container**: The container `VLLMConfig` DOES get `extra="allow"` (per CONTEXT.md), but `ExperimentConfig` retains `extra="forbid"` — the container contract must stay strict.
- **Don't validate passthrough fields**: The whole point is that `model_extra` bypasses our validation. Adding validators that check `model_extra` keys defeats the architecture.
- **Don't iterate model_fields for model_extra**: `model_extra` is separate from `model_fields`. The introspection `get_params_from_model()` correctly uses `model_fields` for typed fields. Passthrough fields never appear in introspection output — this is correct by design.
- **Don't use model_extra for PyTorchConfig passthrough without verifying HuggingFace API surface**: The PyTorchBackend wires to both `from_pretrained()` kwargs AND `model.generate()` kwargs. Extra fields from `model_extra` need routing to the right API. For now, forwarding to `from_pretrained()` is the safe choice.

## Common Pitfalls

### Pitfall 1: BeamSearchParams API difference from SamplingParams

**What goes wrong:** `llm.generate(prompts, BeamSearchParams(...))` may not work the same way as with `SamplingParams`. In vLLM v1, beam search uses a separate `llm.beam_search()` method, not `llm.generate()`.

**Why it happens:** vLLM v1 separated beam search from sampling into a distinct API. `BeamSearchParams` is not a drop-in replacement for `SamplingParams` in `generate()`.

**How to avoid:** Use `llm.beam_search(prompts, BeamSearchParams(...))` when beam search is configured. Add a conditional branch in `_run_measurement()`:
```python
if sampling_params is a BeamSearchParams:
    outputs = llm.beam_search(prompts, sampling_params)
else:
    outputs = llm.generate(prompts, sampling_params)
```

**Warning signs:** `TypeError` or `AttributeError` when passing `BeamSearchParams` to `llm.generate()`.

**Confidence:** MEDIUM — vLLM beam search API confirmed in principle from training data. Verify exact method signature in vLLM docs when running on GPU. The `_build_sampling_params` is currently a `@staticmethod` — it returns the params object; the caller in `_run_measurement` must check the type.

### Pitfall 2: model_extra is empty dict (not None) when no extra fields set

**What goes wrong:** `if engine.model_extra:` correctly handles this — empty dict is falsy. Using `if engine.model_extra is not None:` would call `kwargs.update({})` unnecessarily (harmless but noisy).

**How to avoid:** Use `if engine.model_extra:` (truthy check), not `is not None`.

### Pitfall 3: extra="allow" on PyTorchConfig — routing to correct HuggingFace API

**What goes wrong:** `PyTorchBackend` routes kwargs to two different APIs: `from_pretrained(**model_kwargs)` and `model.generate(**gen_kwargs)`. Naively forwarding all `model_extra` to `from_pretrained()` will break if a user passes generate-time kwargs as extras.

**How to avoid:** For Phase 19.2, forward `pytorch.model_extra` only to `from_pretrained()`. This is consistent — `extra="allow"` is primarily for engine-level params. Generate-time PyTorch params are already exposed via typed fields (temperature, etc. via DecoderConfig). Document this limitation in field descriptions.

**Confidence:** HIGH — `PyTorchBackend._model_load_kwargs()` pattern is clear from the codebase.

### Pitfall 4: kv_cache_memory_bytes wiring name

**What goes wrong:** The vLLM `LLM()` kwarg may not be exactly `kv_cache_memory_bytes`. This is a newer vLLM parameter and naming can vary.

**How to avoid:** The CONTEXT.md specifies the field name as `kv_cache_memory_bytes`. Confirmed usage: wire as `kwargs["kv_cache_memory_bytes"] = engine.kv_cache_memory_bytes`. If vLLM rejects it, the `model_extra` passthrough architecture ensures the user can still set it directly — but they'd also get a validation error from our mutual exclusion validator. Accept this as a low-risk wiring concern; the passthrough architecture is the fallback.

**Confidence:** MEDIUM — field name plausible but not verifiable without live vLLM instance.

### Pitfall 5: VLLMAttentionConfig — flat vs. nested wiring

**What goes wrong:** Attempting to wire `VLLMAttentionConfig` fields as a nested `AttentionConfig` object to `LLM()` will fail. vLLM's `LLM()` constructor does NOT accept an `attention_config=` kwarg; the fields are passed individually as flat kwargs.

**How to avoid:** Wire each field individually using the `_set()` helper, not as a dict or nested object. The `attention_backend` kwarg (from `backend` field) and boolean flags like `use_cudnn_prefill`, `disable_flashinfer_prefill` etc. are direct `LLM()` kwargs.

def _build_llm_kwargs(self, config: ExperimentConfig) -> dict[str, Any]:

            _set("use_prefill_query_quantization", attn.use_prefill_query_quantization)

class TestPassthroughKwargs:

## Open Questions

1. **vLLM beam search API: `llm.beam_search()` vs `llm.generate()`**
   - What we know: vLLM v1 separated `BeamSearchParams` from `SamplingParams`. `BeamSearchParams` class exists.
   - What's unclear: Does `llm.generate(prompts, BeamSearchParams(...))` work, or is it `llm.beam_search(prompts, params)`?
   - Recommendation: Implement both — check `isinstance(sampling_params, BeamSearchParams)` in `_run_measurement()` and branch to the correct method. Use `try/except` fallback if needed. Since no GPU testing in unit tests, use `_FakeBeamSearchParams` to verify kwargs construction; actual API call goes untested in unit tier (same pattern as all vLLM tests).

2. **VLLMAttentionConfig wiring — which fields map to flat `LLM()` kwargs?**
   - What we know: `attention_backend` kwarg is confirmed (used in old `inference_backends/vllm.py` as `attention_backend`). The boolean flags (`use_prefill_decode_attention`, etc.) are vLLM internal options.
   - What's unclear: Whether all 10 boolean flags are accepted as direct `LLM()` kwargs or require some wrapper.
   - Recommendation: Wire all of them using `_set()`. Extra "allow" on `VLLMAttentionConfig` means anything we miss in explicit wiring can still be passed through `model_extra`. No risk of data loss.

3. **`flash_attn_version` and `flash_attn_max_num_splits_for_cuda_graph` — vLLM kwarg names**
   - What we know: These appear in vLLM's `AttentionConfig` internals.
   - What's unclear: Are they accepted as `LLM()` constructor kwargs, or are they internal only?
   - Recommendation: Omit explicit wiring for these two; they'll naturally pass through `model_extra` if a user sets them. If they don't work at the `LLM()` level, that's an upstream concern. Alternatively, note them in field descriptions as "advanced — may require specific vLLM versions."

4. **TensorRTConfig `extra="allow"` — where does `model_extra` route?**
   - What we know: TensorRTConfig gets `extra="allow"` for consistency. TensorRTBackend is not yet active.
   - What's unclear: The TensorRT backend will need a wiring update for `model_extra` when activated in M4.
   - Recommendation: Add `extra="allow"` to TensorRTConfig now. Leave a TODO comment in TensorRTBackend noting that `model_extra` forwarding is needed when M4 activates. No runtime behaviour change — the backend isn't called in M3.
