# Phase 21.1: CI & Versioning Scaffold - Context

**Gathered:** 2026-03-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Migrate CI pipeline from pip to uv with lockfile caching, stabilise the 0.x version scheme, create a fit-for-purpose release workflow with uv build, and integrate Docker image publishing to GHCR. GPU CI strategy and test matrix improvements are out of scope (Phase 22).

</domain>

<decisions>
## Implementation Decisions

### uv lockfile strategy
- Commit `uv.lock` to the repo (uv official recommendation, confirmed for libraries by Charlie Marsh)
- Use `UV_FROZEN: true` as global CI env var (Pydantic's approach — fast, no lockfile validation overhead)
- Use `astral-sh/setup-uv` action with `enable-cache: true` and `cache-suffix` per job
- Set `UV_NO_PROGRESS: '1'` for cleaner CI logs

### CI extras handling
- Standard CI (lint, type-check, test) installs dev dependencies only — no backend extras ([pytorch], [vllm], [tensorrt])
- Backend extras are 2GB+ and require GPU; tested only in GPU CI (Phase 22)
- Disable `gpu-ci.yml` for now — defer GPU CI runner strategy to Phase 22

### Python version matrix
- Keep 3.10 + 3.12 (minimum supported + latest stable)
- Matrix applies to test job only; lint and type-check run on 3.12

### Release workflow
- Trigger: tag push `v*` (existing pattern, matches milestone completion flow)
- CI checks (lint, type-check, test) must pass before release builds
- Build with `uv build` (not `python -m build`)
- Attach wheel + sdist to GitHub Release (since no PyPI yet, this gives users an install path)
- Use `softprops/action-gh-release` with `generate_release_notes: true` + `files: dist/*`

### Docker publish workflow
- Registry: GHCR only (`ghcr.io/{owner}/llenergymeasure`)
- Tags: `v{version}` + `latest` (no CUDA variant tags yet — single CUDA version per image)
- Separate `docker-publish.yml` supporting both:
  - `workflow_call` — called by release workflow (version synced)
  - `workflow_dispatch` — manual trigger for republishing without a release
- Version passed as input parameter

### Version scheme
- Already clean: v0.1.0 through v0.8.0 tags and GitHub Releases in place
- No tag or release cleanup needed
- Ensure release workflow produces clean 0.x releases going forward

### Claude's Discretion
- Exact `setup-uv` version (v5/v6/v7 — use latest stable)
- CI job naming conventions
- Whether to add `uv lock --check` as a separate CI job or pre-commit hook
- Exact Docker build steps and Dockerfile selection for publish
- Cache key structure and `cache-suffix` naming

</decisions>

<specifics>
## Specific Ideas

- Pydantic's CI pattern as reference: `UV_FROZEN: true`, `UV_NO_PROGRESS: '1'`, `uv sync --group dev`
- Ruff's release pattern as reference: `release.yml` calls `build-docker.yml` as reusable workflow
- `uv build --no-sources` before publish to verify package builds without uv-specific overrides
- Release workflow should gate on CI passing (not just trust the tag)

</specifics>

<deferred>
## Deferred Ideas

- PyPI publishing via trusted publishers — add when approaching 1.0 (use `uv publish` with OIDC)
- CUDA variant Docker tags (e.g., `v0.9.0-cuda12.4`) — add when supporting multiple CUDA versions
- Docker Hub as second registry — add if discoverability becomes important
- GitHub GPU runners as alternative to self-hosted — evaluate in Phase 22
- `workflow_dispatch` trigger option for releases — consider if manual gate needed later
- PR label trigger (`ci:gpu`) for GPU CI on PRs — Phase 22

</deferred>
