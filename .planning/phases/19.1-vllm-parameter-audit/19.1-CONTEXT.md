# Phase 19.1: vLLM Parameter Audit - Context

**Gathered:** 2026-03-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Expand `VLLMConfig` to expose energy-relevant vLLM parameters — covering the `LLM()` engine initialisation args and `SamplingParams` fields that affect memory allocation, kernel dispatch, or generation length. Wire params through to the backend, update SSOT introspection with test values and constraint metadata, and extend sweep grammar to support `vllm.engine.X` and `vllm.sampling.X` dotted paths.

Creating posts, Docker image publishing, and full (~60-param) vLLM API parity are separate phases.

</domain>

<decisions>
## Implementation Decisions

### Parameter coverage
- Energy-relevant subset only: params that affect VRAM allocation, kernel dispatch, or generation length (~15–20 params total)
- Full LLM() + SamplingParams parity (~60 params) is deferred — captured as a future todo
- Roadmap examples as starting candidates: `enforce_eager`, `block_size`, `kv_cache_dtype`, `swap_space`, `dtype`, `speculative_model`, `gpu_memory_utilization`, `max_tokens`
- Research step (19.1-01) confirms the final list against upstream vLLM API

### Config structure
- Nested, mirroring vLLM's own two-API separation — design principle is explicitness
- Sub-keys: `engine:` (maps to `LLM()` init args) and `sampling:` (maps to `SamplingParams`)
- Example structure:
  ```yaml
  backend: vllm
  vllm:
    engine:
      dtype: float16
      enforce_eager: true
      gpu_memory_utilization: 0.9
    sampling:
      temperature: 0.0
      max_tokens: 512
  ```

### Validation strictness
- Unknown params: raise a clear error at parse time with a message listing valid params (fail fast — prevents silent misconfiguration in long studies)
- Value constraints: Pydantic validators on `VLLMConfig` fields (e.g. `gpu_memory_utilization` must be in (0.0, 1.0], `block_size` must be a power of 2)
- Cross-field constraints: validate known dependencies (e.g. `speculative_model` set without `num_speculative_tokens` → error)
- Sweep expansion: validate each expanded `ExperimentConfig` at study parse time — researcher sees all errors before any experiment starts

### Sweep grammar
- Sweep paths mirror the nested config structure: `vllm.engine.X` and `vllm.sampling.X`
- Both engine and sampling sweep paths supported in this phase (`max_tokens` is energy-relevant)
- Always backend-scoped — `vllm.engine.dtype` and `pytorch.dtype` are independent sweep axes; no cross-backend aliases
- Bad sweep paths (e.g. typos) fail at study load, not at dispatch — consistent with eager validation
- Example:
  ```yaml
  sweep:
    vllm.engine.block_size: [8, 16, 32]
    vllm.engine.enforce_eager: [true, false]
    vllm.sampling.max_tokens: [256, 512]
  ```

### SSOT introspection
- Test values: valid + one known-invalid value per field (sufficient to exercise validators without excessive maintenance burden)
- No `energy_relevant` flag — all fields treated equally in SSOT; energy relevance is evident from field docstrings
- Constraint metadata included alongside test values: valid ranges (min/max) and allowed string literals per field
- API mapping documented: each SSOT entry records whether the field routes to `LLM()` or `SamplingParams`

### Claude's Discretion
- Exact list of ~15–20 energy-relevant params (research step 19.1-01 determines this)
- Internal wiring strategy for routing `engine` vs `sampling` fields at backend call time
- Pydantic validator implementation details

</decisions>

<specifics>
## Specific Ideas

- Design principle: be explicit — config structure should match vLLM's own API separation so researchers familiar with vLLM see exactly what they expect
- Sweep paths should be unambiguous even if engine and sampling ever share a field name (nested structure prevents collision)
- Phase 4.1 (PyTorch audit) is the reference implementation — follow the same pattern for field definition, backend wiring, and SSOT entries

</specifics>
