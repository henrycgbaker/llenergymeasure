- Unknown params: raise a clear error at parse time with a message listing valid params (fail fast)

- Sweep paths mirror the nested config structure: `vllm.engine.X` and `vllm.sampling.X`

## Summary

The current `VLLMConfig` in `backend_configs.py` is the M1 minimal schema with only 5 fields: `max_num_seqs`, `tensor_parallel_size`, `gpu_memory_utilization`, `enable_prefix_caching`, `quantization`. The production `VLLMBackend` in `core/backends/vllm.py` already wires 5 of these correctly. There is a large gap: the CONTEXT.md defines ~15–20 energy-relevant params spread across vLLM's two APIs (engine init and sampling), none of which are currently in `VLLMConfig`.

The CONTEXT.md mandates a **nested** `VLLMConfig` structure with `engine:` and `sampling:` sub-objects mirroring vLLM's API separation. This is a structural change from the current flat `VLLMConfig`. The sweep grammar currently splits on the first dot (`vllm.X` → backend=`vllm`, param=`X`), which handles one level of nesting. The `vllm.engine.X` sweep paths require the sweep expansion to handle two levels of nesting properly — this is achievable by treating `engine.block_size` as a nested dict key in the raw config, which Pydantic's nested model parsing then resolves correctly.

The SSOT introspection system (`config/introspection.py`) auto-discovers fields from Pydantic models, so adding nested sub-models extends it automatically — but `get_backend_specific_params()`, `get_mutual_exclusions()`, `get_param_skip_conditions()`, and `get_validation_rules()` need manual additions for the new cross-validators and skip conditions.

**Primary recommendation:** Define `VLLMEngineConfig` and `VLLMSamplingConfig` as nested Pydantic models within `VLLMConfig`, add ~15 fields total across both, update `_build_llm_kwargs()` and `_build_sampling_params()` in `core/backends/vllm.py` to wire new fields, and update introspection for constraint metadata and skip conditions. The sweep grammar requires no code changes — it already handles two-level nesting via `split(".", 1)`.

## Current State: Gap Analysis

### `VLLMConfig` (M1 minimal — as-built)

Source: `src/llenergymeasure/config/backend_configs.py`

| Field | Type | Wired in Backend? |
|-------|------|-------------------|
| `max_num_seqs` | `int \| None` | YES — `_build_llm_kwargs()` |
| `tensor_parallel_size` | `int \| None` | YES |
| `gpu_memory_utilization` | `float \| None` | YES |
| `enable_prefix_caching` | `bool \| None` | YES |
| `quantization` | `Literal[...] \| None` | YES |

**Gap:** The CONTEXT.md target list includes ~13 additional fields not yet in `VLLMConfig`. None of the `SamplingParams`-specific fields (other than what's already in `DecoderConfig`) are exposed for vLLM-specific override.

### Sweep Grammar: Current Behaviour vs Required

Current `_expand_sweep()` in `study/grid.py` handles `vllm.X`:
```python
backend_name, param = key.split(".", 1)  # "vllm.engine.block_size" → ("vllm", "engine.block_size")
scoped_dims.setdefault("vllm", {})["engine.block_size"] = values
# → config_dict["vllm"]["engine.block_size"] = value
```

This produces `config["vllm"] = {"engine.block_size": 16}`. When Pydantic parses `VLLMConfig(engine={"block_size": 16})` — the nested dict form — it works correctly IF `VLLMConfig` has an `engine` field typed as `VLLMEngineConfig | None`.

**Key insight:** The current `split(".", 1)` implementation already handles three-segment paths (`vllm.engine.block_size`) correctly for nested Pydantic models. No changes to the sweep grammar are needed. When `param = "engine.block_size"`, the config dict gets `{"engine": {"block_size": 16}}` — wait, this is not what happens. The current code does `config_dict["vllm"]["engine.block_size"] = 16`, NOT `config_dict["vllm"]["engine"]["block_size"] = 16`.

**Correction:** The current sweep code produces `config["vllm"] = {"engine.block_size": 16}` (a literal dotted key, NOT nested). Pydantic `VLLMConfig(engine.block_size=16)` would reject this because `engine.block_size` is not a valid field name and `extra="forbid"` is set.

**Therefore:** The sweep grammar DOES need a change to handle three-segment paths. When `vllm.engine.block_size` is in the sweep, the raw config must have `{"vllm": {"engine": {"block_size": 16}}}`, not `{"vllm": {"engine.block_size": 16}}`.

The `_expand_sweep()` function needs to recursively unflatten the backend-scoped params. OR the `_unflatten()` helper already in `loader.py` (which handles dotted keys recursively via `split(".", 1)`) can be applied to the backend-scoped dict. The cleanest fix is to apply `_unflatten()` to the backend-scoped dict before merging.

## Architecture Patterns

### Recommended Config Structure

Following the CONTEXT.md mandate for nested structure mirroring vLLM's API separation:

```
backend_configs.py:
├── VLLMEngineConfig     # Maps to vllm.LLM() constructor args
├── VLLMSamplingConfig   # Maps to vllm.SamplingParams() vLLM-specific args
└── VLLMConfig           # Container: engine: VLLMEngineConfig, sampling: VLLMSamplingConfig
```

### Pattern 1: Nested VLLMConfig

**What:** Split `VLLMConfig` into `VLLMEngineConfig` + `VLLMSamplingConfig` sub-models, both nested inside a new `VLLMConfig`.

**Why:** Mirrors vLLM's own API separation. `LLM()` engine args and `SamplingParams` args are distinct APIs at runtime. Sweep paths `vllm.engine.X` and `vllm.sampling.X` become unambiguous.

**Example:**
```python
# Source: Phase 4.1 PyTorch audit pattern (backend_configs.py)

class VLLMEngineConfig(BaseModel):
    """vLLM LLM() constructor args — engine-level configuration."""
    model_config = {"extra": "forbid"}

    # Memory management
    gpu_memory_utilization: float | None = Field(default=None, ge=0.0, lt=1.0,
        description="GPU memory fraction for KV cache (None -> 0.9)")
    swap_space: float | None = Field(default=None, ge=0.0,
        description="CPU swap space in GiB for KV cache offloading (None -> 4)")
    cpu_offload_gb: float | None = Field(default=None, ge=0.0,
        description="CPU RAM to offload model weights to (None -> 0, disabled)")
    block_size: int | None = Field(default=None,
        description="KV cache block size in tokens: 8, 16, or 32 (None -> 16)")
    kv_cache_dtype: Literal["auto", "fp8", "fp8_e5m2", "fp8_e4m3"] | None = Field(
        default=None, description="KV cache storage dtype (None -> auto = model dtype)")

    # Execution
    enforce_eager: bool | None = Field(default=None,
        description="Disable CUDA graphs, always use eager mode (None -> False)")
    enable_chunked_prefill: bool | None = Field(default=None,
        description="Chunk large prefills across multiple scheduler iterations (None -> False)")

    # Batching
    max_num_seqs: int | None = Field(default=None, ge=1,
        description="Max concurrent sequences per iteration (None -> 256)")
    max_num_batched_tokens: int | None = Field(default=None, ge=1,
        description="Max tokens processed per scheduler iteration (None -> auto)")
    max_model_len: int | None = Field(default=None, ge=1,
        description="Max sequence length (input + output). Overrides model config (None -> model default)")

    # Parallelism
    tensor_parallel_size: int | None = Field(default=None, ge=1,
        description="Tensor parallel degree (None -> 1)")
    pipeline_parallel_size: int | None = Field(default=None, ge=1,
        description="Pipeline parallel stages (None -> 1)")

    # KV cache optimization
    enable_prefix_caching: bool | None = Field(default=None,
        description="Automatic prefix caching (None -> False)")

    # Quantization
    quantization: Literal["awq", "gptq", "fp8", "marlin", "bitsandbytes"] | None = Field(
        default=None, description="Quantization method. Requires pre-quantized model.")

    # Speculative decoding
    speculative_model: str | None = Field(default=None,
        description="Draft model for speculative decoding (HF model name or path)")
    num_speculative_tokens: int | None = Field(default=None, ge=1,
        description="Tokens to draft per speculative step (required with speculative_model)")

    @model_validator(mode="after")
    def validate_speculative_decoding(self) -> "VLLMEngineConfig":
        """speculative_model requires num_speculative_tokens."""
        if self.speculative_model is not None and self.num_speculative_tokens is None:
            raise ValueError(
                "speculative_model requires num_speculative_tokens to be set"
            )
        return self

    @model_validator(mode="after")
    def validate_block_size(self) -> "VLLMEngineConfig":
        """block_size must be 8, 16, or 32."""
        if self.block_size is not None and self.block_size not in (8, 16, 32):
            raise ValueError(
                f"block_size must be 8, 16, or 32, got {self.block_size}"
            )
        return self


class VLLMSamplingConfig(BaseModel):
    """vLLM SamplingParams args — vLLM-specific sampling extensions.

    Universal sampling params (temperature, top_p, top_k, repetition_penalty, min_p)
    live in DecoderConfig and are shared across all backends.
    This model holds vLLM-specific overrides that don't exist in DecoderConfig.
    """
    model_config = {"extra": "forbid"}

    max_tokens: int | None = Field(default=None, ge=1,
        description="Max output tokens, overrides ExperimentConfig.max_output_tokens for this backend (None -> use max_output_tokens)")
    min_tokens: int | None = Field(default=None, ge=0,
        description="Minimum output tokens (None -> 0, no minimum)")
    presence_penalty: float | None = Field(default=None, ge=-2.0, le=2.0,
        description="Presence penalty: penalizes tokens that appear at all (None -> 0.0)")
    frequency_penalty: float | None = Field(default=None, ge=-2.0, le=2.0,
        description="Frequency penalty: penalizes tokens proportional to frequency (None -> 0.0)")
    ignore_eos: bool | None = Field(default=None,
        description="Continue generating past EOS token (None -> False)")


class VLLMConfig(BaseModel):
    """vLLM backend configuration — nested engine + sampling sections."""
    model_config = {"extra": "forbid"}

    engine: VLLMEngineConfig | None = Field(default=None,
        description="Engine-level configuration (vllm.LLM() constructor args)")
    sampling: VLLMSamplingConfig | None = Field(default=None,
        description="Sampling configuration (vllm.SamplingParams args, vLLM-specific only)")
```

### Pattern 2: Sweep Grammar Fix for Three-Segment Paths

**What:** The sweep expansion in `study/grid.py` handles `vllm.engine.block_size` as backend=`vllm`, param=`engine.block_size`. Currently this produces `config["vllm"]["engine.block_size"]` which Pydantic rejects. Fix by recursively unflattening the backend-scoped dict.

**Current (broken for three-segment paths):**
```python
# line 347 in grid.py
config_dict.setdefault(backend, {})[dim_key] = value
# Result: config["vllm"]["engine.block_size"] = 16  ← Pydantic rejects
```

**Fixed approach:**
```python
# Apply _unflatten() from loader.py to the backend-scoped value being set
from llenergymeasure.config.loader import _unflatten  # or duplicate logic inline

# When building config_dict for a sweep point:
if dim_key in backend_scoped:
    # Unflatten nested paths: "engine.block_size" → {"engine": {"block_size": value}}
    backend_dict = config_dict.setdefault(backend, {})
    nested = _unflatten({dim_key: value})
    config_dict[backend] = deep_merge(backend_dict, nested)
```

This produces `config["vllm"] = {"engine": {"block_size": 16}}`, which Pydantic correctly parses as `VLLMConfig(engine=VLLMEngineConfig(block_size=16))`.

**Alternative:** Keep the sweep grammar change minimal. Instead of importing from loader, inline the two-level unflatten logic directly in `_expand_sweep()` for the backend-scoped dict.

### Pattern 3: Backend Wiring for Nested Config

**What:** `_build_llm_kwargs()` and `_build_sampling_params()` in `core/backends/vllm.py` must read from the new nested structure.

**Example:**
```python
# In VLLMBackend._build_llm_kwargs():
vllm_cfg = config.vllm
if vllm_cfg is not None:
    engine = vllm_cfg.engine
    if engine is not None:
        if engine.gpu_memory_utilization is not None:
            kwargs["gpu_memory_utilization"] = engine.gpu_memory_utilization
        if engine.enforce_eager is not None:
            kwargs["enforce_eager"] = engine.enforce_eager
        if engine.block_size is not None:
            kwargs["block_size"] = engine.block_size
        if engine.kv_cache_dtype is not None:
            kwargs["kv_cache_dtype"] = engine.kv_cache_dtype
        if engine.swap_space is not None:
            kwargs["swap_space"] = engine.swap_space
        if engine.speculative_model is not None:
            kwargs["speculative_config"] = {
                "model": engine.speculative_model,
                "num_speculative_tokens": engine.num_speculative_tokens,
            }
        # ... etc for all engine fields

# In VLLMBackend._build_sampling_params():
if vllm_cfg is not None:
    sampling = vllm_cfg.sampling
    if sampling is not None:
        if sampling.max_tokens is not None:
            params["max_tokens"] = sampling.max_tokens  # Override ExperimentConfig.max_output_tokens
        if sampling.presence_penalty is not None:
            params["presence_penalty"] = sampling.presence_penalty
        if sampling.frequency_penalty is not None:
            params["frequency_penalty"] = sampling.frequency_penalty
        if sampling.min_tokens is not None:
            params["min_tokens"] = sampling.min_tokens
        if sampling.ignore_eos is not None:
            params["ignore_eos"] = sampling.ignore_eos
```

### Pattern 4: SSOT Introspection Updates

**What:** The SSOT system (`config/introspection.py`) auto-discovers fields via `get_params_from_model()`. Nested models are recursed automatically (the `include_nested=True` flag + `hasattr(annotation, "model_fields")` check). So `get_backend_params("vllm")` will automatically discover `vllm.engine.gpu_memory_utilization` etc. once the nested models exist.

**Manual updates still required:**

1. `get_backend_specific_params()["vllm"]` — add new field paths
2. `get_mutual_exclusions()` — add speculative decoding cross-validator entry
3. `get_param_skip_conditions()` — add skip conditions for GPU-requiring params
4. `get_validation_rules()` — add new cross-validator documentation
5. `_get_custom_test_values()` — add known-invalid test values per field

**Key SSOT requirement from CONTEXT.md:**
- Valid test values per field (auto-derived from constraints by introspection)
- ONE known-invalid value per field (manually specified in `_get_custom_test_values()`)
- Constraint metadata (ge/le/options — auto-derived from Pydantic Field metadata)
- API mapping: LLM() vs SamplingParams (document in field descriptions or add API routing info to SSOT)

### Anti-Patterns to Avoid

- **Flat VLLMConfig:** Adding all fields to a single flat `VLLMConfig` (not nested) — violates the CONTEXT.md locked decision, breaks `vllm.engine.X` sweep paths
- **Checking for non-None in wiring code:** The new nested `VLLMConfig` means `config.vllm.engine` can be None separately from `config.vllm`. Always null-check both levels.
- **Hardcoded non-default checks:** Old backend code had `if vllm_cfg.gpu_memory_utilization != 0.9:` — WRONG. Use `if field is not None:` consistently (None-as-default pattern).
- **Migrating existing flat fields:** The M1 minimal `VLLMConfig` had `max_num_seqs`, `tensor_parallel_size`, `gpu_memory_utilization`, `enable_prefix_caching`, `quantization` at the flat level. These must move into `engine:` sub-config. This is a breaking change to YAML configs using the old flat structure — document this, but M3 is pre-release so breaking changes are acceptable.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Dotted path unflattening | Custom string splitting in sweep | `_unflatten()` in `loader.py` + `deep_merge()` | Already handles recursive dotted keys; tested |
| Pydantic cross-validators | Custom validation logic outside models | `@model_validator(mode="after")` on `VLLMEngineConfig` | Pydantic integrates with schema validation and SSOT |

## Common Pitfalls

### Pitfall 1: Flat Config Migration Not Complete
**What goes wrong:** Existing flat `VLLMConfig` fields (`max_num_seqs`, `tensor_parallel_size`, `gpu_memory_utilization`, `enable_prefix_caching`, `quantization`) are removed from the top-level `VLLMConfig` but the old backend wiring in `_build_llm_kwargs()` still reads `config.vllm.tensor_parallel_size`.
**Why it happens:** Both the Pydantic model and backend wiring code need updating simultaneously.
**How to avoid:** Update both `backend_configs.py` (schema) and `core/backends/vllm.py` (wiring) in the same plan task, or immediately sequential tasks.
**Warning signs:** `AttributeError: VLLMConfig has no field 'tensor_parallel_size'` at runtime.

### Pitfall 2: Sweep Grammar Three-Segment Paths Not Fixed
**What goes wrong:** Sweep `vllm.engine.block_size: [8, 16, 32]` produces raw config `{"vllm": {"engine.block_size": 8}}`. Pydantic rejects `VLLMConfig(engine.block_size=8)` with `extra=forbid`.
**Why it happens:** Current `_expand_sweep()` only unflattens the backend-level — not the nested param within the backend.
**How to avoid:** Apply `_unflatten()` to the backend-scoped param dict before merging into `config_dict[backend]`.
**Warning signs:** Study load raises `ConfigError: all generated config(s) are invalid` with a Pydantic validation error mentioning the dotted key.

### Pitfall 3: `config.vllm.tensor_parallel_size` Access After Migration
**What goes wrong:** The `test_vllm_backend.py` tests and `validate_config()` method in `core/backends/vllm.py` access `vllm_cfg.tensor_parallel_size` directly, which will break after moving to `vllm_cfg.engine.tensor_parallel_size`.
**How to avoid:** Search for all `vllm_cfg.` accesses and update them. Run existing unit tests to catch breakage early.
**Warning signs:** `AttributeError` or `ValidationError` in existing tests.

### Pitfall 4: SSOT Introspection Misses Nested Fields
**What goes wrong:** `get_backend_specific_params()["vllm"]` hardcodes a flat list of `"vllm.max_num_seqs"` etc. After migration, these paths change to `"vllm.engine.max_num_seqs"`.
**Why it happens:** `get_backend_specific_params()` is manually maintained, not auto-derived.
**How to avoid:** Update `get_backend_specific_params()["vllm"]` to use the new nested paths. Also verify that `get_params_from_model()` correctly recurses into `VLLMEngineConfig` and `VLLMSamplingConfig` (it does, via the `include_nested=True` logic).
**Warning signs:** `get_backend_params("vllm")` returns stale or missing paths.

### Pitfall 5: SSOT Auto-Discovery for Nested Models
**What goes wrong:** `get_params_from_model(VLLMConfig, prefix="vllm")` should auto-discover `VLLMEngineConfig` fields with paths like `vllm.engine.gpu_memory_utilization`. The existing `get_params_from_model()` handles nested Pydantic models via `hasattr(annotation, "model_fields")` check (line 163 in introspection.py). This WILL work for `VLLMEngineConfig` because it is a BaseModel subclass.
**How to verify:** `python3 -c "from llenergymeasure.config.introspection import get_backend_params; print(list(get_backend_params('vllm').keys()))"` — should show nested paths after the change.

### Pitfall 6: `vllm.gpu_memory_utilization` vs `vllm.engine.gpu_memory_utilization` in Existing Tests
**What goes wrong:** `test_vllm_backend.py` creates `VLLMConfig(gpu_memory_utilization=0.8)` which will break after restructuring.
**How to avoid:** Update existing tests to use `VLLMConfig(engine=VLLMEngineConfig(gpu_memory_utilization=0.8))`.

### Pitfall 7: `speculative_model` in vLLM v1
**What goes wrong:** vLLM v0.6+ changed speculative decoding config format. The `LLM()` arg is `speculative_config={"model": ..., "num_speculative_tokens": ...}` not a direct `speculative_model=` kwarg.
**Confidence:** MEDIUM — confirmed in `core/inference_backends/vllm.py` (the v1.x backend) which uses `speculative_config` dict.

### Updated Backend Wiring
```python
# Source: core/backends/vllm.py
def _build_llm_kwargs(self, config: ExperimentConfig) -> dict[str, Any]:
    kwargs: dict[str, Any] = {
        "model": config.model,
        "dtype": self._map_precision(config.precision),
        "trust_remote_code": True,
        "seed": config.random_seed,
    }

    vllm_cfg = config.vllm
    if vllm_cfg is not None and vllm_cfg.engine is not None:
        engine = vllm_cfg.engine
        _set_if_not_none = lambda k, v: kwargs.update({k: v}) if v is not None else None
        _set_if_not_none("gpu_memory_utilization", engine.gpu_memory_utilization)
        _set_if_not_none("swap_space", engine.swap_space)
        _set_if_not_none("cpu_offload_gb", engine.cpu_offload_gb)
        _set_if_not_none("block_size", engine.block_size)
        _set_if_not_none("kv_cache_dtype", engine.kv_cache_dtype)
        _set_if_not_none("enforce_eager", engine.enforce_eager)
        _set_if_not_none("enable_chunked_prefill", engine.enable_chunked_prefill)
        _set_if_not_none("max_num_seqs", engine.max_num_seqs)
        _set_if_not_none("max_num_batched_tokens", engine.max_num_batched_tokens)
        _set_if_not_none("max_model_len", engine.max_model_len)
        _set_if_not_none("tensor_parallel_size", engine.tensor_parallel_size)
        _set_if_not_none("pipeline_parallel_size", engine.pipeline_parallel_size)
        _set_if_not_none("enable_prefix_caching", engine.enable_prefix_caching)
        _set_if_not_none("quantization", engine.quantization)

        if engine.speculative_model is not None:
            kwargs["speculative_config"] = {
                "model": engine.speculative_model,
                "num_speculative_tokens": engine.num_speculative_tokens,
            }

    return kwargs
```

## Open Questions

1. **`block_size` constraint: Pydantic or validator?**
   - What we know: vLLM accepts 8, 16, or 32. A `Literal[8, 16, 32]` type would be cleaner than a `@model_validator`.
   - What's unclear: Do newer vLLM versions accept other values (e.g., 64)?
   - Recommendation: Use `Literal[8, 16, 32] | None` for explicit constraint documentation. If vLLM expands to 64, we add it then.

2. **`quantization` Literal values — current set?**
   - What we know: M1 minimal has `Literal["awq", "gptq", "fp8"]`. vLLM also supports `marlin`, `bitsandbytes`, `gguf`, `squeezellm`, `compressed-tensors`.
   - Recommendation: Expand to `Literal["awq", "gptq", "fp8", "fp8_e5m2", "fp8_e4m3", "marlin", "bitsandbytes"]`. Leave `gguf` and others for the full-parity phase.
   - Confidence: MEDIUM — verify against `vllm.config.QuantizationConfig` source.

3. **`max_tokens` in VLLMSamplingConfig vs ExperimentConfig.max_output_tokens**
   - What we know: `ExperimentConfig.max_output_tokens` already sets the generation length universally. `vllm.sampling.max_tokens` would override it for vLLM specifically.
   - What's unclear: Is this override semantics clearly documented enough to avoid confusion?
   - Recommendation: Include `max_tokens` in `VLLMSamplingConfig` with description "Overrides ExperimentConfig.max_output_tokens for vLLM sweeps". This allows `vllm.sampling.max_tokens: [256, 512]` sweep while keeping `max_output_tokens: 128` as default for other backends.

4. **Backward compatibility with old flat VLLMConfig YAML syntax**
   - What we know: Any YAML with `vllm: gpu_memory_utilization: 0.9` (flat) will break with the new nested schema.
   - Risk assessment: Low — vLLM backend is not yet in production use (it's being activated in M3). No external users have production configs with `vllm:` sections.
   - Recommendation: No migration path needed. Document the new structure in CONTEXT.md and YAML reference.
