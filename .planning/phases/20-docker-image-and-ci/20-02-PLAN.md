**Complete workflow content:**

```yaml
name: Docker Publish

on:
  push:
    tags: ["v*"]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    strategy:
      fail-fast: false
      matrix:
        include:
          - backend: vllm
            dockerfile: docker/Dockerfile.vllm
          - backend: pytorch
            dockerfile: docker/Dockerfile.pytorch
          # Future milestones add entries here:
          # - backend: tensorrt
          #   dockerfile: docker/Dockerfile.tensorrt
          # - backend: sglang
          #   dockerfile: docker/Dockerfile.sglang

    steps:
      - uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Login to GHCR
        if: github.event_name != 'pull_request'
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Extract metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ghcr.io/henrycgbaker/llenergymeasure/${{ matrix.backend }}
          flavor: |
            latest=auto
            suffix=-cuda12
          tags: |
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}

      - name: Build and push
        uses: docker/build-push-action@v6
        with:
          context: .
          file: ${{ matrix.dockerfile }}
          push: ${{ github.event_name != 'pull_request' }}
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha,scope=${{ matrix.backend }}
          cache-to: type=gha,mode=max,scope=${{ matrix.backend }}
```

**Design decisions reflected in this workflow:**

1. **Separate workflow** — CONTEXT.md decision: not merged into release.yml. Docker builds are slow; independent failure.

2. **Trigger:** `push: tags: ["v*"]` for publishing, `pull_request: branches: [main]` for validation.

3. **Matrix strategy:**
   - `fail-fast: false` — if vLLM build fails (huge image), PyTorch build should still complete.
   - Adding a backend = one new entry in `matrix.include` list.
   - Commented-out entries for TensorRT (M4) and SGLang (M5) show future extension point.

4. **GHCR login:** Only on tag push (not PRs). Uses `GITHUB_TOKEN` (no PAT needed).

5. **Metadata action:**
   - `images: ghcr.io/henrycgbaker/llenergymeasure/${{ matrix.backend }}` — matches CONTEXT.md registry path.
   - `flavor: latest=auto` — attaches `latest` only on default branch tags (prevents pre-release overwrite).
   - `flavor: suffix=-cuda12` — all tags get `-cuda12` suffix (CUDA 12 only per CONTEXT.md).
   - `type=semver,pattern={{version}}` — produces e.g. `1.19.0-cuda12`.
   - `type=semver,pattern={{major}}.{{minor}}` — produces e.g. `1.19-cuda12` (minor version pin).

6. **Build and push:**
   - `push: ${{ github.event_name != 'pull_request' }}` — build-only on PRs, build+push on tags.
   - `context: .` — repo root (Dockerfiles COPY from root).
   - `file: ${{ matrix.dockerfile }}` — per-backend Dockerfile.
   - `cache-from/cache-to: type=gha,scope=${{ matrix.backend }}` — per-backend cache scope prevents collisions.

**Tag output examples:**

For tag `v1.19.0` pushed from main:
- `ghcr.io/henrycgbaker/llenergymeasure/vllm:1.19.0-cuda12`
- `ghcr.io/henrycgbaker/llenergymeasure/vllm:1.19-cuda12`
- `ghcr.io/henrycgbaker/llenergymeasure/vllm:latest`
- `ghcr.io/henrycgbaker/llenergymeasure/pytorch:1.19.0-cuda12`
- `ghcr.io/henrycgbaker/llenergymeasure/pytorch:1.19-cuda12`
- `ghcr.io/henrycgbaker/llenergymeasure/pytorch:latest`

**Do NOT:**
- Modify ci.yml or release.yml
- Add GPU runners (builds don't need GPU — they just package the image)
- Add manual workflow_dispatch trigger (not requested)
- Use `docker build` + `docker push` shell commands (use the official actions)
- Use `fail-fast: true` (one backend failure should not cancel others)
- Omit the `scope` parameter on cache (would cause cache collisions between backends)
