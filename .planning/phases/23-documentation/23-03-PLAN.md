    **CONTEXT.md decision:** Docker is the recommended/default path. This guide is a full walkthrough, not just a link to NVIDIA docs. vLLM-focused with mention of future backends.

    Sections (in order):
    1. **Title: Docker Setup Guide** — Brief intro: llenergymeasure uses Docker to run vLLM (and future TensorRT-LLM, SGLang) backends in isolated containers with GPU access. This guide walks through setting up Docker with GPU support from scratch.
    2. **Prerequisites** — Linux host, NVIDIA GPU with drivers installed, `nvidia-smi` working. Note: Docker GPU passthrough is Linux-only (no macOS/Windows GPU support in Docker).
    3. **Step 1: Install Docker** — Brief: link to Docker's official install guide for Ubuntu/Debian. Show the post-install step: `sudo usermod -aG docker $USER` and `newgrp docker`. Verify: `docker run hello-world`.
    4. **Step 2: Install NVIDIA Drivers** — Brief: if `nvidia-smi` already works, skip. Otherwise link to NVIDIA's driver install guide. Show verification: `nvidia-smi` output.
    5. **Step 3: Install NVIDIA Container Toolkit** — Full commands for Ubuntu/Debian (the primary target):
       ```bash
       curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg
       curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list | \
         sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | \
         sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list
       sudo apt-get update
       sudo apt-get install -y nvidia-container-toolkit
       sudo nvidia-ctk runtime configure --runtime=docker
       sudo systemctl restart docker
       ```
       Note: for other distros, link to NVIDIA's install guide.
    6. **Step 4: Verify GPU Access in Docker** — `docker run --rm --gpus all nvidia/cuda:12.4.1-base-ubuntu22.04 nvidia-smi`. Explain expected output. This is the same check that `llem`'s Docker pre-flight performs.
    7. **Step 5: Verify with llenergymeasure** — `llem config` should now show the GPU. Then: `llem run --model gpt2 --backend vllm` with a study YAML that sets `runners: {vllm: docker}`. Or the simpler: create a YAML file and run it.
    8. **Image Management** — Explain the Docker image naming: `ghcr.io/henrycgbaker/llenergymeasure/vllm:{version}-cuda{cuda_major}`. Explain that images are pulled automatically on first run. Mention `docker pull` for pre-fetching. Mention future backends: TensorRT-LLM and SGLang images will follow the same pattern.
    9. **Docker-specific Troubleshooting** — Common issues:
       - "docker: Error response from daemon: could not select device driver" → NVIDIA CT not installed or Docker not restarted
       - "nvidia-container-cli: initialization error" → driver version mismatch
       - Permission denied → user not in docker group
       - GPU not visible in container → `--gpus all` flag missing (llem handles this automatically)
       - Shared memory errors with vLLM → llem sets `--shm-size 8g` automatically
       - Pre-flight check failures → explain each check from `docker_preflight.py` (Docker CLI, NVIDIA CT binary, host nvidia-smi, GPU visibility, CUDA/driver compat)

    Tone: step-by-step imperative. Assume NO Docker knowledge. Each step has a verification command. Source Docker pre-flight error messages from `src/llenergymeasure/infra/docker_preflight.py` for the troubleshooting section.

       ```yaml
       backend: vllm
       runners:
         vllm: docker
       ```
       Show config with vLLM-specific options using the NESTED structure (critical — Phase 19.1 decision):
       ```yaml
       backend: vllm
       runners:
         vllm: docker
       vllm:
         engine:
           enforce_eager: false
           gpu_memory_utilization: 0.9
           block_size: 16
         sampling:
           max_tokens: 128
       ```
       List key vLLM engine parameters (enforce_eager, gpu_memory_utilization, block_size, kv_cache_dtype, swap_space, dtype, speculative_model, num_speculative_tokens, etc.).     5. **Switching Between Backends** — Show how to run the same model with different backends:
       ```yaml
       # PyTorch (local)
       model: gpt2
       backend: pytorch

       # vLLM (Docker)
       model: gpt2
       backend: vllm
       runners:
         vllm: docker
       ```
       Explain: changing `backend:` switches the inference engine. vLLM requires Docker runner. PyTorch runs locally by default.
    6. **Runner Configuration** — Explain the runner system:
       - Per-study YAML: `runners: {vllm: docker}` or `runners: {vllm: "docker:custom/image:tag"}`
