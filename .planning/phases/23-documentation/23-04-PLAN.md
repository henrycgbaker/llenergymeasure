    **energy-measurement.md** — What we measure (energy, throughput, FLOPs) and how. Technical depth for researchers.

    Sections:
    1. **Title: Energy Measurement** — What llenergymeasure measures and how it works.
    2. **What We Measure** — Three categories:
       - **Energy (Joules):** GPU power draw integrated over inference time. Primary metric. Measured via NVIDIA Management Library (NVML) polling at high frequency. Optional: Zeus (kernel-level), CodeCarbon (broader system estimate).
       - **Throughput:** Tokens per second (output tokens / inference time). Both total throughput and per-prompt throughput.
       - **FLOPs:** Floating point operations estimated from model architecture (parameter count, sequence lengths). Approximate — based on transformer FLOPs formula, not hardware counters.
    3. **Energy Measurement Backends** — The `energy:` config section:
       - `auto` (default): uses NVML if available, falls back gracefully
       - `nvml`: NVIDIA Management Library. Polls GPU power draw. Most accurate for GPU-only measurement. No additional install needed (pynvml bundled).
       - `zeus`: Zeus energy monitoring. Kernel-level GPU energy measurement. Requires `pip install "llenergymeasure[zeus]"`.
       - `codecarbon`: CodeCarbon tracker. Estimates total system energy (CPU + GPU + RAM). Requires `pip install "llenergymeasure[codecarbon]"`.
       - `null`: Disable energy measurement (throughput-only mode).
    4. **How Energy is Measured** — Technical detail:
       - NVML poller runs in a separate thread during inference
       - Samples GPU power draw (watts) at regular intervals (configurable, default ~100ms)
       - Integrates power over time to compute energy in Joules
       - Reports per-prompt and total energy
       - Timeseries data available in result JSON for detailed analysis
    5. **Peak Memory** — `inference_memory_mb` in results: peak GPU memory used during inference only (not model loading). Measured as the delta between pre-inference and peak NVML memory readings.
    6. **Limitations** — Energy measurement limitations: NVML measures GPU only (not CPU/RAM unless CodeCarbon used), polling frequency limits precision for very short inferences, multi-GPU measurement requires aggregation across devices.

    Source energy measurement implementation from `src/llenergymeasure/core/energy.py` and `src/llenergymeasure/core/measurement.py`. Read these files to confirm poller details.

    **methodology.md** — Statistical approach, warmup, thermal stabilisation, baseline.

    Sections:
    1. **Title: Measurement Methodology** — How llenergymeasure ensures reproducible, reliable energy measurements.
    2. **Warmup** — The `warmup:` config section:
       - Purpose: GPU thermal state and driver caches affect first-run measurements. Warmup runs discard initial measurements.
       - `enabled: true` (default) — runs warmup inferences before measurement
       - `n_warmup: 3` (default) — number of warmup inferences
       - `thermal_floor_seconds: 30` (default) — minimum wait time for GPU temperature to stabilise after warmup
       - Recommendation: always enable for publication-quality measurements. Disable for quick testing.
    3. **Baseline Power** — The `baseline:` config section:
       - Purpose: measures idle GPU power draw before inference to compute delta energy (inference-only energy).
       - `enabled: true` (default) — captures baseline power reading
       - `duration_seconds: 5` (default) — how long to sample baseline power
       - Result: `baseline_power_watts` in output, used to compute `inference_energy_joules` (total minus baseline).
    4. **Multi-Cycle Execution** — Why run multiple cycles:
       - Single measurements have variance (thermal drift, system load, caching)
       - Multiple cycles (n_cycles=3 default in CLI) produce per-cycle results for statistical analysis
       - Cycle ordering (sequential, interleaved, shuffled) controls execution order to reduce systematic bias
       - Shuffled ordering (CLI default) with reproducible seed ensures results are comparable across runs
    5. **Thermal Management** — Thermal gaps between experiments:
       - By default, llenergymeasure inserts thermal gaps between experiments in a study
       - Purpose: allow GPU to return to baseline temperature, reducing thermal carry-over between configurations
       - `--no-gaps` flag disables this (for speed, at cost of measurement quality)
    6. **Reproducibility** — How to maximise reproducibility:
       - Use fixed random_seed (default: 42)
       - Use shuffled cycle ordering with n_cycles >= 3
       - Enable warmup and baseline
       - Report full configuration (llem stores effective_config in results)
       - Control for system load (dedicated GPU, no other processes)

    Source warmup/baseline/thermal details from `src/llenergymeasure/config/models.py` (WarmupConfig, BaselineConfig) and `src/llenergymeasure/study/runner.py` for thermal gap logic.
