  <action>
    Add a new job `docs-freshness` to `.github/workflows/ci.yml`. This job:
    1. Checks out the repo
    2. Sets up uv (same pattern as other jobs: `astral-sh/setup-uv@v7`, `enable-cache: true`, `cache-suffix: docs-3.12`, `python-version: "3.12"`)
    3. Runs `uv sync --dev` to install dependencies
    4. Runs the generation scripts that can work without GPU:
       - `uv run python scripts/generate_config_docs.py --output docs/generated-config-reference.md` (or wherever the committed output lives — see note below)
       - `uv run python scripts/generate_cli_reference.py --output docs/generated-cli-reference.md`
       - `uv run python scripts/generate_invalid_combos_doc.py` (only if it can run without GPU test results — check if it imports introspection module which should work without GPU)
    5. Runs `git diff --exit-code docs/` to fail if any generated output differs from committed version.

    **Note on output paths:** The CONTEXT.md says generated content is inlined into parent docs, not in a separate generated/ folder. However, for CI freshness checking, the scripts need to write to a known path so `git diff` can compare. Two approaches:
    - Option A: Scripts write to temporary files, compare against the relevant section extracted from the parent doc.
    - Option B: Scripts write standalone files to `docs/` with names like `_generated-config-reference.md` (prefixed with underscore to indicate generated), and the parent docs include them via a manual copy step.

    **Use Option B** for simplicity: scripts output standalone generated files. The parent topic docs (study-config.md, cli-reference.md, troubleshooting.md) will include a note like `<!-- See scripts/generate_*.py for regeneration -->` and the generated content will be pasted in by the doc-writing plans. The CI freshness check verifies the standalone generated files match what the scripts produce.

    Actually, simpler: since CONTEXT.md says inline, just have the scripts write to stdout and the CI step pipes to a temp file and diffs against a known section marker in the parent doc. But this is fragile.

    **Simplest approach:** Have the scripts write to `docs/_generated/` (underscore prefix, gitignored — no, needs to be committed for diff). Write to `docs/_generated/config-reference.md`, `docs/_generated/cli-reference.md`, `docs/_generated/invalid-combos.md`. The parent docs manually paste this content. CI regenerates these files and diffs.

    Wait — CONTEXT.md explicitly says "Not in a separate generated/ folder". So the generated files should not exist as separate files. The content must be inlined.

    **Final approach:** The CI freshness check runs the scripts, captures their stdout, and compares against markers in the parent docs. Use a wrapper script `scripts/check_docs_freshness.sh`:
    ```bash
    #!/usr/bin/env bash
    set -euo pipefail
    # Regenerate and compare
    uv run python scripts/generate_config_docs.py > /tmp/config-ref.md
    uv run python scripts/generate_cli_reference.py > /tmp/cli-ref.md
    # Extract sections from committed docs using markers
    # Compare with diff
    ```

    This is too complex for a CI step. **Pragmatic approach:** Have the scripts write to well-known paths. The CI step regenerates and checks `git diff --exit-code` on those paths. The parent docs will contain an HTML comment marker pointing to the script but the actual content is maintained by running the scripts and committing the output. This satisfies "inline" (content is in the doc, not a separate file) while being CI-checkable.

    **Implementation:** Add a `docs-freshness` job that:
    1. Runs `uv run python scripts/generate_config_docs.py --check` — the script itself compares its output against the relevant section in `docs/study-config.md` (between `<!-- BEGIN CONFIG REFERENCE -->` and `<!-- END CONFIG REFERENCE -->` markers) and exits non-zero if different.
    2. Similarly for cli-reference and invalid-combos.

    If the `--check` mode is too complex for Task 1's scripts, simplify: just run the scripts and ensure they exit 0 (no runtime errors). The actual content correctness is a manual review concern. Add a comment in CI noting that full freshness checking requires marker-based extraction (future improvement).

    **Keep it simple for M3:** The CI job runs the three generation scripts and verifies they exit without error. This catches import breakage and model schema changes that would cause script failures. Full marker-based diffing is a future improvement.

    ```yaml
    docs-freshness:
      runs-on: ubuntu-latest
      steps:
        - uses: actions/checkout@v4
        - uses: astral-sh/setup-uv@v7
          with:
            enable-cache: true
            cache-suffix: docs-3.12

            uv run python scripts/generate_config_docs.py > /dev/null
            uv run python scripts/generate_cli_reference.py > /dev/null
            uv run python scripts/generate_invalid_combos_doc.py > /dev/null
    ```

    The `--extra pytorch` is needed because the config models import backend configs which may reference PyTorch types. Check if base install is sufficient — if generate_config_docs.py only uses `model_json_schema()` (schema extraction, no runtime instantiation), base install may work. Test and adjust.
